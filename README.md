# S7_to_OPCUA：西门子 S7 至 OPC UA 协议网关

![version](https://img.shields.io/badge/version-0.1.1-blue) [![Docker Hub](https://img.shields.io/badge/Docker%20Hub-impxssive%2Fs7--opcua-2496ED?logo=docker&logoColor=white)](https://hub.docker.com/r/impxssive/s7-opcua) ![platforms](https://img.shields.io/badge/platforms-linux%2Famd64%20%7C%20arm64-555) ![C11](https://img.shields.io/badge/C-11-00599C?logo=c&logoColor=white)

本项目是一个基于 C 语言开发的工业通信网关，旨在实现西门子 S7 系列 PLC（包含 S7-300/400/1200/1500）数据的高效采集，并将其转换为标准 **OPC UA** 节点，以便于 SCADA、MES 及各类 OPC UA 客户端系统（如 UaExpert）进行数据读取。

本网关底层基于以下两大开源组件构建：
- **[snap7](https://snap7.sourceforge.net/)**（libsnap7）：用于实现 S7 通信协议的客户端功能。
- **[open62541](https://www.open62541.org/)**：用于提供 OPC UA 服务端功能。

> **项目说明**：当前为初始版本（主要支持只读模式），兼具教学与工程实践价值。源码包含详尽注释，涵盖工业通信协议转换、字节序处理、多线程并发管理以及基于回调的服务器架构等核心知识点。

## 1. 项目背景与核心功能

工业现场的底层设备（如西门子 PLC）通常采用私有的通信协议（S7 协议），而上层应用系统（如 SCADA、MES）多倾向于采用国际标准协议（OPC UA）。本项目充当协议转换的中间件枢纽，消除异构系统间的通信壁垒。

```text
  西门子 PLC          [ 协议网关 ]                上位机系统
 ┌─────────┐  S7协议  ┌──────────────┐  OPC UA  ┌───────────┐
 │ DB/M/I/Q │ ───────► │ 数据采集与缓存│ ───────► │ UaExpert  │
 │ 寄存器数据│         │ 协议转换与暴露│         │ SCADA/MES │
 └─────────┘         └──────────────┘         └───────────┘
```

## 2. 架构设计与数据流转

本系统采用**“按需采集机制 (On-Demand Polling) 结合基于 TTL 的缓存失效策略”**。系统在无客户端访问时处于休眠状态（零负载）；仅当 OPC UA 客户端发起读取请求且当前缓存数据已逾期时，才触发对 PLC 的物理读取。

```text
                         ┌────────────────────────────────────────────┐
   物理 PLC              │                 网关进程                     │          上位机
 ┌──────────┐           │   ┌─────────┐         ┌──────────┐           │       ┌──────────┐
 │   PLC    │◄─S7读取─ ┤   │ Tag缓存 │◄──读取───│  OPC UA  │           │◄OPCUA─│ 客户端   │
 │          │ (仅缓存失效)│   │(值+TTL) │──下发缓存►│  Server  │  读取请求 │ 请求  │          │
 └──────────┘           │   └─────────┘         └──────────┘           │       └──────────┘
                        └────────────────────────────────────────────┘
                          基于回调机制驱动底层数据采集，避免无效的后台轮询
```

**请求处理与数据路由逻辑：**

1. **缓存命中**：若缓存时间戳未超出预设的 TTL（Time-To-Live），则直接返回内存缓存数据，**避免触发物理网络 IO**。
2. **缓存失效 / 首次读取**：若缓存已过期或尚未初始化，网关通过 Snap7 接口发起 S7 读请求，更新本地缓存并返回最新数据。
3. **静默状态**：在无外部客户端接入的场景下，网关与 PLC 之间维持零通信交互，最大化降低底层设备负载。

**设计考量：**

- **设备保护**：高频轮询易导致 PLC 通信资源枯竭。引入按需采集策略配合 TTL，有效过滤瞬时高频读取，实现防抖与削峰。
- **缓存有效性 (TTL)**：用户可通过配置 `cache_ttl_ms` 参数平衡数据实时性与底层设备负载。
- **动态回调驱动**：基于 open62541 的 DataSource 机制，实现节点数据的动态响应（Lazy Evaluation），有效区分内部初始化读取与真实客户端请求。

## 4. 核心模块解析

系统遵循自底向上的严格分层架构，具备高内聚低耦合特性，支持独立模块化测试验证。

### ① 数据解析层 (s7_types)

负责网络层大端序 (Big-Endian) 原始字节流向标准 C 语言数据结构的转换。
- `s7_decode(type, buf, bit, &out)`：多态解析函数。
- 全面支持 S7 标准内建类型体系：`BOOL` / `BYTE` / `SINT` / `USINT` / `INT` / `UINT` / `WORD` / `DINT` / `UDINT` / `DWORD` / `REAL` / `LREAL`。

### ② 配置管理层 (config)

基于 cJSON 实现 `gateway.json` 结构的加载与强类型映射。
- 具备严格的语义校验能力：数据类型匹配、边界条件审查（如 DB 号限制、位偏移约束等），支持快速失败（Fail-fast）策略。

### ③ 物理通信层 (s7_client)

将 Snap7 库复杂接口抽象为面向对象的 `S7Conn` 句柄模型。
- 支持自动重连机制与异常状态隔离。
- 封装 `Cli_ReadArea` 操作，并联动数据解析层完成一次性读取与反序列化。

### ④ 数据缓存层 (tag_cache)

基于 Pthreads 的高性能、线程安全的数据状态容器。
- 复合数据槽设计：存储 `数值 + 通信质量 (Quality) + 时间戳 (Timestamp)`。
- 读写分离机制：利用互斥锁 (Mutex) 实现数据快照隔离，保障读写事务的原子性。核心提供 `get_fresh` 接口响应按需采集决策。

### ⑤ 协议暴露层 (opcua_server)

构建 OPC UA 地址空间，映射配置点表至 DataSource 变量节点。
- 构建标准的系统层次结构：`Objects / <DeviceName> / <TagName>`。
- 自动化质量代码映射：正常状态映射为 `Good`，离线状态映射为 `BadNoCommunication`，严格遵循 OPC UA 规范。

### ⑥ 服务主进程 (main)

管理网关生命周期：
1. 资源初始化（配置加载、分配缓存区）
2. 延迟连接建立（Lazy Connection）
3. 启动 OPC UA 主事件循环
4. 捕获系统终止信号并执行资源回收（Graceful Shutdown）

## 5. 关键技术概念

| 概念名称 | 技术内涵 | 相关模块 |
|---|---|---|
| **大端序 (Big-Endian)** | S7 协议基于大端字节序，异构 PC 架构（如 x86/ARM 小端序）必须执行端序转换。 | `s7_types.c` |
| **位级提取 (Bit Masking)** | `BOOL` 类型在字节序列中的位偏移计算。 | `s7_types.c` |
| **内存对齐与越界保护** | 使用 `memcpy` 处理浮点数与大整数反序列化，规避强制指针转换导致的总线错误。 | `s7_types.c` |
| **不透明指针 (Opaque Pointer)** | 利用 C 语言机制实现封装，隐藏内部实现细节以提升 ABI 稳定性。 | `s7_client.h` |
| **DataSource 回调机制** | 延迟计算模式（Lazy Evaluation），推迟数据访问至真正需要时刻。 | `opcua_server.c` |
| **缓存过期控制 (TTL)** | 分布式系统中常用的请求合并与限流机制，用于隔离物理层带宽抖动。 | `tag_cache.c` |
| **异常保留机制** | 工业通信标准实践：通信中断时挂起旧值并降级数据质量（如标记为 Bad），禁止置零。 | `tag_cache.c` |

### 附录：Snap7 编译与环境集成（仅限初次构建）

由于 Homebrew 暂不提供针对 macOS 的预编译包，本项目中包含经过适配的编译产出物。若需重建底层库环境，请参照以下流程：

```bash
# 1. 获取 Snap7 核心源码包
curl -L -o /tmp/snap7.7z "https://master.dl.sourceforge.net/project/snap7/1.4.2/snap7-full-1.4.2.7z?viasf=1"

# 2. 执行交叉编译构建动态链接库（移除 Linux 专属的 -lrt 标识）
cd /tmp/snap7-full-1.4.2/src
clang++ -O3 -fPIC -shared -Isys -Icore -Ilib \
  sys/*.cpp core/*.cpp lib/*.cpp -lpthread -o libsnap7.dylib

# 3. 集成部署至项目结构
cp libsnap7.dylib  <ProjectRoot>/third_party/snap7/lib/
cp ../release/Wrappers/c-cpp/snap7.h  <ProjectRoot>/third_party/snap7/include/
install_name_tool -id @rpath/libsnap7.dylib  <ProjectRoot>/third_party/snap7/lib/libsnap7.dylib
```

## 8. 配置参数详解

网关的运行行为由 `config/gateway.json` 统一管理。开发者需根据实际拓扑配置设备通信节点和数据表。

```jsonc
{
  "opcua": { "port": 4840 },               // [可选] OPC UA 监听端口，缺省为 4840
  "collection": { "cache_ttl_ms": 1000 },  // [可选] 缓存时效周期(ms)，缺省为 1000
  "plcs": [                                // 物理 PLC 组网清单
    {
      "name": "Line1_PLC",                 // [必填] PLC 标识 (构成 NodeId 的 Namespace)
      "ip": "192.168.0.1",                 // [必填] 目标 IPv4 地址
      "port": 102,                         // [可选] ISO-on-TCP 端口，缺省为 102
      "rack": 0,                           // [可选] 硬件机架号，缺省为 0
      "slot": 1,                           // [可选] CPU 槽位号，缺省为 1
      "tags": [                            // 采集变量点位清单
        { "name": "Motor1.Speed", "area": "DB", "db": 10, "start": 0,  "type": "REAL" },
        { "name": "Motor1.Run",   "area": "DB", "db": 10, "start": 4,  "bit": 0, "type": "BOOL" },
        { "name": "Motor1.Count", "area": "DB", "db": 10, "start": 6,  "type": "DINT" }
      ]
    }
  ]
}
```

### 点位 (Tag) 描述规范

| 属性名 | 语义定义 | 合规取值范围 | 是否必填 |
|---|---|---|---|
| `name` | 数据点标识符 | 有效的英数字符串 | ✅ |
| `area` | 访问存储区 | `DB` (数据块), `M` (位存储区), `I` (输入), `Q` (输出) | ✅ |
| `db` | DB 模块编号 | 正整数 (仅针对 `DB` 区域) | `DB` 时必填 |
| `start` | 字节地址偏移量 | 非负整数 | ✅ |
| `bit` | 位地址偏移量 | `0` ~ `7` (仅针对 `BOOL` 字段) | `BOOL` 时必填 |
| `type` | 标量数据类型 | 见下文类型映射体系规范表 | ✅ |
| `node_id` | 显式 OPC UA NodeId（覆盖自动生成设定） | 标准 NodeId 格式，如 `ns=2;s=[1001001]` | 可选（详见第 16 章） |

> **配置说明**：`node_id` 为可选字段，旨在提供向后兼容性与灵活对接能力。若配置此字段，网关将按照指定的 NodeId 暴露节点，以便与现场工程点表或上位机系统的配置无缝对接；若省略该字段，系统将采用默认规则自动生成格式为 `ns=1;s=PLC名称.点位名称` 的 NodeId。

### 类型映射体系规范

| 配置文件类型 | S7 架构对等物 | 内存占用 | OPC UA 类型模型映射 | 地址映射实例 |
|---|---|---|---|---|
| `BOOL`  | Boolean        | 1 Bit   | `Boolean` | `DB10.DBX4.0` |
| `BYTE`  | Unsigned 8-bit | 1 Byte  | `Byte` (UInt8) | `DB10.DBB0` |
| `USINT` | Unsigned 8-bit | 1 Byte  | `Byte` (UInt8) | `DB10.DBB0` |
| `SINT`  | Signed 8-bit   | 1 Byte  | `SByte` (Int8) | `DB10.DBB0` |
| `WORD`  | Unsigned 16-bit| 2 Bytes | `UInt16` | `DB10.DBW0` |
| `UINT`  | Unsigned 16-bit| 2 Bytes | `UInt16` | `DB10.DBW0` |
| `INT`   | Signed 16-bit  | 2 Bytes | `Int16`  | `DB10.DBW0` |
| `DWORD` | Unsigned 32-bit| 4 Bytes | `UInt32` | `DB10.DBD0` |
| `UDINT` | Unsigned 32-bit| 4 Bytes | `UInt32` | `DB10.DBD0` |
| `DINT`  | Signed 32-bit  | 4 Bytes | `Int32`  | `DB10.DBD0` |
| `REAL`  | IEEE 754 Float | 4 Bytes | `Float`  | `DB10.DBD0` |
| `LREAL` | IEEE 754 Double| 8 Bytes | `Double` | `DB10.DBD0` |

## 9. 运行指南

启动网关进程：

```bash
./gateway                       # 加载缺省配置：config/gateway.json
./gateway /path/to/custom.json  # 载入指定路径的配置文件
```

系统运行时标准输出：

```text
[系统] 配置加载成功: config/gateway.json (PLC 接入数: 1, 绑定端口: 4840, 缓存TTL: 1000ms)
[系统] 网关初始化完毕(按需采集)。监听地址: opc.tcp://localhost:4840 (终止信号: SIGINT/Ctrl+C)
[系统] 提示: 无客户端请求时不会访问 PLC。
```

> **运行机制提示**：基于系统采用的**按需采集**与**延迟连接 (Lazy Connection)** 机制，网关在启动初始化阶段**不会**主动建立与 PLC 的通信链路。当且仅当 OPC UA 服务端接收到客户端读取请求，且对应节点的本地缓存已过期失效时，系统才会触发底层的网络连接与数据拉取操作。因此，服务启动早期的日志输出中默认不包含 PLC 连接状态信息，相关的连接确认（如 `[FakeLine] 已连接 ...`）将在实际数据交互发生时记录。

系统全面支持平滑退出（Graceful Shutdown），通过捕获终止信号以确保进程终结时资源的完整回收与清理。

### 虚拟仿真模式接入

本工程内置 `fake_plc` 仿真工具组件，旨在提供可重现的测试信号与动态模拟的过程数据，从而降低开发与联调阶段对物理设备硬件环境的依赖。

1. **启动仿真环境**（拉起仿真 PLC 进程，默认绑定至本地 1102 端口）：
   `make fake_plc && ./fake_plc`
2. **启动网关服务**（加载针对仿真环境优化的配置文件）：
   `make gateway && ./gateway config/fake_plc.json`
3. **启动客户端监控工具**（周期性监测节点数据状态）：
   `make ua_monitor && ./ua_monitor config/fake_plc.json [刷新间隔秒数]`

   `ua_monitor` 指令支持传入可选的**刷新间隔参数（秒）**，支持浮点数值设定，最小允许值为 0.1 秒，缺省参数下为 1 秒：

   ```bash
   ./ua_monitor config/fake_plc.json 0.5   # 设定每 0.5 秒发起一次读取请求
   ./ua_monitor config/fake_plc.json 5     # 设定每 5 秒发起一次读取请求
   ./ua_monitor config/fake_plc.json       # 采用默认配置，每 1 秒发起一次读取请求
   ```

> **系统频率概念解析**：在系统运行过程中，存在两类相互独立的控制频率。
> - **客户端轮询频率**：由 `ua_monitor` 启动参数控制，代表外部系统或客户端向网关发起数据读取请求的频次。
> - **物理采集限制频率 (TTL)**：由网关配置文件中的 `collection.cache_ttl_ms` 决定，代表网关允许向底层 PLC 发起真实网络 I/O 请求的最高频次限制。
>
> **底层负载评估**：网关对物理 PLC 的实际读取频率受制于上述两者的共同约束，且由其中**较低的频率**起决定性作用。例如：若客户端配置为每 0.5 秒读取一次，而网关 TTL 阈值设定为 3 秒，则网关系统仍仅每 3 秒触发一次物理 S7 读取操作，客户端在此期间内的其他高频请求将直接由网关的内存快照进行响应。**合理配置 TTL 参数是保障底层 PLC 稳定运行、避免通信过载的核心控制手段**。

## 12. 硬件设备接入指南（重要）

对于部署在生产环境中的物理西门子控制器（S7-1200 / S7-1500 系列），必须在 TIA Portal（博图）工程软件中完成以下通信配置：

1. **启用 PUT/GET 通信许可**：
   在设备视图中进入 CPU 属性配置，导航至 “保护与安全 (Protection & Security) -> 连接机制 (Connection mechanisms)”，确保已勾选 “允许来自远程伙伴（PLC、HMI、OPC、...）的 PUT/GET 通信访问 (Permit access with PUT/GET communication from remote partner)”。
2. **关闭数据块的优化访问属性**：
   本系统基于传统的绝对地址映射机制进行数据采集。因此，针对需对外暴露的过程数据所在的 DB 数据块，必须进入该数据块的属性配置界面，并取消勾选“优化的块访问 (Optimized block access)”选项。
3. **架构寻址参数配置**：
   网关配置文件中的 `rack`（机架号）与 `slot`（槽号）参数需与物理设备的硬件组态严格一致。常规配置基准如下：
   - 针对 S7-1200 / S7-1500 体系架构，常规参数为：`rack = 0, slot = 1`。
   - 针对 S7-300 / S7-400 体系架构，常规参数为：`rack = 0, slot = 2`。

## 13. 故障排查建议

在实施部署与调试过程中，若遇系统运行异常，可参考下述排查策略：

| 异常现象表现 | 诊断思路与排查措施 |
|---|---|
| `Connection refused` (网络连接被拒) | 表明底层 TCP 连接未能建立。需排查目标设备 IP 地址的连通性、PLC 设备的供电运行状态，并确认 TCP 端口 102 未被防火墙拦截。 |
| 读取数值漂移或数据解析异常 | 通常由数据地址映射偏差或类型长度不匹配引发。需优先确认 TIA Portal 工程中目标 DB 块已明确取消“优化的块访问”设定；同时重新审核配置表中的字节地址和位地址偏移量。 |
| 拒绝访问 / 协议层级通讯错误 | 表明网络层连接正常，但 S7 协议栈层面拒绝请求。常见原因系 PLC 控制器硬件配置中尚未开启 PUT/GET 协议通信权限，请进入工程组态软件中激活。 |
| 布尔型数据（BOOL）状态恒定为 false | 系位数据掩码提取逻辑异常所致。需复核 JSON 配置项中 `start`（字节偏移）与 `bit`（位偏移，合规范围 0-7）参数是否正确。 |
| 进程启动报错：OPC UA 端口冲突 | 表示配置的监听端口（默认 4840）已被其他网络进程抢占。建议使用系统命令（如 `lsof -i :4840`）定位占用进程并关闭，或通过修改配置文件中的 `opcua.port` 字段重定向服务端口。 |

项目已发布预构建的**多架构 Docker 镜像**（`linux/amd64` + `linux/arm64`），可覆盖 x86 架构服务器、ARM 架构边缘设备（如树莓派系列）以及 Apple Silicon 的 macOS 平台。镜像开箱即用，目标机器无需安装任何编译器或依赖。

镜像地址：[`impxssive/s7-opcua`](https://hub.docker.com/r/impxssive/s7-opcua)（Docker Hub）。

拉取镜像：

```bash
docker pull impxssive/s7-opcua:0.1.1
```

镜像内部已集成 **Excel 点表自动转换工具**（Python 3 运行时 + `openpyxl` + `tools/xlsx_to_config.py`），入口脚本支持两种启动方式：

```bash
# 方式一：直接加载 JSON 配置启动（常规模式）
# 通过 -v 将宿主机的配置目录挂载至容器的 /app/config
docker run --rm -p 4840:4840 -v /宿主机/配置目录:/app/config \
  impxssive/s7-opcua:0.1.1 config/你的配置.json

# 方式二：直接传入 Excel 点表，容器内自动完成转换后再启动网关
docker run --rm -p 4840:4840 -v /宿主机/配置目录:/app/config \
  impxssive/s7-opcua:0.1.1 --xlsx /app/config/点表.xlsx \
  --ip 192.168.0.10 --plc-name 炉区PLC
```

> **说明**：
> - `--xlsx` 后的首个参数为 Excel 文件路径；其余参数（`--ip`、`--port`、`--plc-name`、`--opcua-port`、`--cache-ttl-ms` 等）将原样转发给内部转换脚本（详见第 16 章）。处理中文点表时必须提供 `--ip`。
> - 端口：`-p 4840` 用于对外暴露 OPC UA 服务端口；网关作为客户端向 PLC 出站连接 102 端口，使用 Docker 默认桥接网络即可。
> - 生产环境建议使用**明确的版本号**（如 `0.1.1`）而非 `latest`，以保证部署可复现。

## 16. Excel 点表导入与自动解析工具

在工业控制系统集成实施阶段，现场测点的数据映射关系主要依附于 Excel 电子表格形式进行管理与交付，而本系统网关底层引擎依赖标准化的 JSON 数据结构作为配置输入。为消弭这一工程实施断层，项目生态中特别提供了一款**解耦设计的 Python 自动化转换脚本工具** `tools/xlsx_to_config.py`，专职负责将 Excel 格式的点表数据无损转化为网关所需的 JSON 配置描述集：

```text
现场 Excel 点表源数据文件(.xlsx)  ──►  xlsx_to_config.py 转换解析脚本  ──►  结构化 JSON 配置文件  ──►  网关核心进程加载
```

> **工具边界申明**：此脚本被定义为一套辅助性的独立工具集，网关的 C 语言主服务进程在任何阶段均不会对其发起调用，系统运行态不产生对其的强依赖。运行该脚本前，需确保宿主系统环境已完备安装 `openpyxl` 依赖模块。针对面临 PEP 668 环境限制策略的 macOS 操作系统用户，推荐采用项目内置的 Python 虚拟环境执行流程：
> 首先构建环境：`python3 -m venv .venv && ./.venv/bin/python3 -m pip install openpyxl`
> 其次调用脚本：`./.venv/bin/python3 tools/xlsx_to_config.py <参数列表>`

转换工具具备自适应格式感知能力，能够识别**两种**典型的数据表格体裁，主要依据表格头部特征执行智能匹配（同理允许用户通过 `--format cn|simple` 命令行参数执行强制体裁声明）。

### 16.1 工业标准体裁格式（cn 格式标准）

此数据格式深度契合国内主流工业现场实施习惯：具备标准化的中文表头信息描述、沿用西门子官方推崇的绝对地址寻址表示法，并兼具自定义 NodeID 规范配置能力。具体数据列的解析映射关系定义如下：

| Excel 列命名规范 | JSON 配置文件映射字段 | 解析与处理策略说明 |
|---|---|---|
| `点位名称` | 映射至 tag 对象内的 `name` 属性（即测点唯一标识符） | 此为强制性约束字段；若处理过程中发生值缺失，解析逻辑将自动回退提取 `点位ID` 字段数值作为备选命名填补。 |
| `点位地址` | 聚合解析为 `area`、`db`、`start` 以及 `bit` 四项属性集合 | 系统核心内嵌基于西门子绝对地址表示法的智能解析引擎，具体解析规则及边界限制详阅下述第 16.1.1 章节说明。 |
| `数据类型` | 映射至 tag 对象内的 `type` 属性 | 将依照规范映射为网关底层预设的通用数据类型定义，且此属性将作为计算底层内存区块读取宽度的唯一依据准则，详阅下述第 16.1.2 章节说明。 |
| `NodeID` | 映射至 tag 对象内的 `node_id` 属性 | 解析工具对该数据项执行无损透传处理至输出文件，网关引擎后续将遵循该自定义设定部署并暴露目标 OPC UA 节点（相关机制详阅下述第 16.4 章节说明）。 |
| `点位ID` | 辅测点标识备份字段 | 处于休眠状态，仅当主标识 `点位名称` 字段出现空值缺失时，该字段信息方会被激活并抽调作为补充替代标识项。 |
| `单位` / `最新数据` / `更新时间` / `设备号` | 系统执行主动过滤及忽略动作 | 业务逻辑判定此类信息归属为系统的运行时状态属性参数，剔除在静态配置文件的生成逻辑链条之外。 |

**底层网络连接信息注入机制**：鉴于此种规范格式的点表本身不具备表述底层控制器网络寻址配置（例如 IP 地址等）的能力，相关连接配置必需借助转换工具运行时的命令行参数进行外部干预注入（设计约定：单一独立配置文件表征与单台物理 PLC 设备的通信拓扑关系）：

```bash
./.venv/bin/python3 tools/xlsx_to_config.py TargetPoints.xlsx -o config/output.json \
    --ip 192.168.0.10 --port 102 --rack 0 --slot 1 --plc-name TargetPLC \
    --opcua-port 4840 --cache-ttl-ms 1000
```

> **命令行参数调度说明**：在针对 `cn` 模式进行解析作业时，`--ip` 参数被设定为必须显式提供的必填项；系统对其他通信参数配置项进行了默认值赋权处理，例如：`--port` 参数缺省值为 102，`--rack` 参数缺省值为 0，`--slot` 参数缺省值为 1，`--plc-name` 标识缺省值为 PLC1。

#### 16.1.1 地址语法解析规则 → 转化为 area / start / bit 集合属性（基于西门子绝对地址规范体系）

| 规范语法格式样例 | 系统解析引出结果集合 | 底层内存区块寻址含义映射 |
|---|---|---|
| `DB1.DBX8.0` | 提取 `area=DB`, `db=1`, `start=8`, 兼得 **`bit=0`** | 定位指向位于数据块 DB 内的布尔离散型量数据点 (BOOL) |
| `DB1.DBB2` | 提取 `area=DB`, `db=1`, `start=2` | 定位指向位于数据块 DB 内起始偏移为 2 的单字节区块 |
| `DB1.DBW2` | 提取 `area=DB`, `db=1`, `start=2` | 定位指向位于数据块 DB 内起始偏移为 2 的字数据类型区块 (Word, 占用 2 字节空间) |
| `DB1.DBD0` | 提取 `area=DB`, `db=1`, `start=0` | 定位指向位于数据块 DB 内起始偏移为 0 的双字数据类型区块 (Double Word, 占用 4 字节空间) |
| `M10.0` / `MB10` / `MW10` / `MD10` | 统一提取 `area=M`, 基准偏移 `start=10`（若为位偏移书写则附带 `bit` 值） | 映射访问内部位存储映像区域 (Memory Marker 标志寄存器区域) |
| `I0.0` / `IB0` / `IW0` / `ID0` | 统一提取标识为 `area=I` | 映射访问外部物理输入过程映像区域 |
| `Q0.0` / `QB0` / `QW0` / `QD0` | 统一提取标识为 `area=Q` | 映射访问外部物理输出过程映像区域 |

> **解析逻辑机制及容错策略申明**：转换引擎由地址语法字符串（特定标识如 `X/B/W/D`）执行**起始字节内存地址偏移量**数据的抽取动作；但对于**实际触发底层内存区块连续读取的字节宽度基准，系统被设计为严格从属于并遵守「数据类型」单元列下的设定约束**。由此，倘若在配置数据录入时发生了数据类型定义要求同地址书写标识宽度内涵不一致的冲突情景（例如：将数据类型单元定义成占用 4 字节的 `float32`，而在地址单元格内却录入表征 2 字节宽度的 `DB1.DBW2` 描述），转换脚本将作非阻断式兼容处理：通过标准输出流投递一条警示级别消息进行提醒通告，并不中断主处理流程，继而系统将强制根据既定数据类型规定的字节长度（此时按 4 字节标准）完成内存读取参数设定。出于规避潜在不可预期工程风险的考量，强烈建议在实施文档的编制环节采取更为严谨地道的地址映射书写范式（例如就此例而言，正规书写体应矫正为 `DB1.DBD2`）。

#### 16.1.2 预置数据类型映射准则（数据类型列声明内容 → 关联转换至网关系统规范类型）

转换引擎处理模块具备大小写字符不敏感的智能匹配机制，现阶段可支持对如下类别常用工程声明体裁的判别响应：

| Excel 表格单元格填入类型标识体裁 | 内部映射采用之网关基准数据类型 | 理论内存区块空间占用基数 | 与 OPC UA 国际规范标准数据模型之桥接关系 |
|---|---|---|---|
| `bool` / `bit` | BOOL | 占据 1 个离散位 | 映射至 `Boolean` 类型体系 |
| `int8` / `sint` | SINT | 占据 1 字节 | 映射至 `SByte` 类型体系 |
| `byte` / `uint8` / `usint` | BYTE | 占据 1 字节 | 映射至 `Byte` 类型体系 |
| `int16` / `int` | INT | 占据 2 字节 | 映射至 `Int16` 类型体系 |
| `word` / `uint16` / `uint` | WORD | 占据 2 字节 | 映射至 `UInt16` 类型体系 |
| `int32` / `dint` | DINT | 占据 4 字节 | 映射至 `Int32` 类型体系 |
| `dword` / `uint32` / `udint` | DWORD | 占据 4 字节 | 映射至 `UInt32` 类型体系 |
| `float32` / `real` / `float` | REAL | 占据 4 字节 | 映射至 `Float` 类型体系 |
| `float64` / `lreal` / `double` | LREAL | 占据 8 字节 | 映射至 `Double` 类型体系 |

> **边界异常应对处置预案**：一旦系统引擎捕捉探查到表格数据列中出现不包含于上述预置清单范围内的未登录标识书写体，转换脚本将引发执行过程的阻断异常状态，并将包含未知类型的原始内容片段列印至终端予以通告，用以引导并督促操作人员回溯至 Excel 点表源文件中完成配置纠偏修复工作。

### 16.2 显式属性映射点表格式体系（simple 格式标准）

假定项目实际维系的业务点表被设计为将全部所需参数细拆分列配置、实现各字段值直接平铺的排布模式（从而彻底免除需介入基于正则或语法分析的地址解构环节），系统同样提供针对该 `simple` 模式的支持。在此模式设定下，系统要求输入的工作表格首行（表头部分）需依照如下严格规定的顺序及命名标准进行构造排版：
`plc_name ip port rack slot poll_interval_ms tag_name area db start bit type`

置于此种格式框架之下，表格内的各数据列被赋予与网关底层 JSON 配置体系中对应属性字段维持绝对无缝映射的特征纽带。其中，底层网络通信连接相关的参数数值将被系统直接由对应的行级单元数据中提炼（由此可免去借由命令行传入 `--ip` 参数的必要），并且在机制上深度兼容并许可于单一数据工作表文件内集中承载跨越多台异构 PLC 设备的混合测点配置资源集（系统逻辑基于对 `plc_name` 字段值的甄别达成自动的分组聚合梳理操作）。

### 16.3 转换系统异常全盘扫描校验及退出执行状态码规范

基于全方位改善工程部署与维护阶段错误排查体验与效率的宗旨，配套的格式转换工具模块被深度植入了一套实现全链路闭环的数据健全度校验机制策略。在转换进程的演进周期中，脚本程序将被强制执行一次针对源数据流的无遗漏全局完整性扫描作业；待全案审查程序完结之后，所有触发校验断言规则的异常及报错项将被实施统一聚合梳理，随后于执行终端一并进行总揽展示。此种批处理集中反馈模式可有效规避系统遭遇单一错误即刻产生中断挂起的低效逐次纠错短板。

相关的安全校验策略集被设计为与底层核心网关服务程序的 `config.c` 配置规则检验逻辑模块保持深度同源对接互操作，涉及范围宽泛覆盖但不限于：针对数据存储分配区标识符语义合法性的判定验证、确保涉密 `DB` 数据操作必含合法映射标号的强依赖校验、针对 `BOOL` 离散位提取操作确保关联挂载必选项 `bit (限定域跨度 0~7)` 指向地址偏移量有效性的合法验证、关键实体对象名称及网络 IP 地址字符跨度超限风险的边界管控审查、单台汇聚物理从站下属关联参数是否维持内在逻辑自洽的一致性约束、同一受控区域拓扑内部署各测点标识命名的去重唯一性冲突检查等。

系统执行完毕后的规范化退出状态码标识反馈释义规则设定如下：
- 反馈执行码值 `0`：指示当前格式重组与编译转换任务链条执行无误并圆满完结。
- 反馈执行码值 `1`：标识被读取的数据点表**在内容语义维度层面**探测查证到有违规的逻辑冲突或未遵循格式规范的现象发生（系统控制终端界面将进行逐项追踪溯源的详细异常警示清册输出展示，需依据提示指引实施表格内容整改动作后发起源头数据的二重装载运转）。
- 反馈执行码值 `2`：通告在系统运行时遭遇了导致整体流程无法正常推进的**平台级严重异常故障事件**（诱发肇因涵盖如：存在严重的核心功能数据列架构缺损崩塌情况、目标底层数据文件被宿主安全体系施加严格权限阻断无法顺利调阅访问、运行基础环境侦测核验认定关联模块组件库（如 `openpyxl` 等）缺失未能装配、指定承载数据的特定工作表 (sheet) 命名标称错误造成加载锚定失败或是输入的运行启动参数传递解析被识别存在非法的逻辑瑕疵等各类不可逾越因素所导致）。

### 16.4 基于自定义 NodeID 规范实现节点寻址暴露机制支持

得益于新版配置文件结构的演进引入的独立 `node_id` 配置属性支持段（配置体例如：`ns=2;s=[1001001]`），此项扩容极大地丰富并且能够直接驱动、精确指导网关进程节点于部署投产后的 OPC UA 全局地址映射空间当中的发布暴露展现策略。这一底层特性的增强支持直接回应并达成了契合满足下游复杂上位系统（如 SCADA 组态软件调度中心）依据特定既定规范施加的系统化节点信息读取约束。在多源异构交织的复合型网络工业通讯环境中，这一特性能完美构筑**零阻力平滑应用系统代际迁移与网关系统级无感无缝热替换能力**的基础支撑设施。

配置解析生效应用策略行为范式：
- 若在解析目标配置文件期间判定探查到对应子节点已明确**被显式挂载声明了** `node_id` 数据属性，系统底层网关守护进程将以强制执行的优先级指令采信此预先给定的 NodeId 属性值为依据凭证，对该目标通信节点进行构造与组网发布操作（此过程中所涉之未曾建档的诸如 `ns=2` 一类 Namespace 命名空间声明亦将被系统内建组件自动化发起并托管完成注销登记环节）。
- 反之，如若确认在目标节点的配置描述清单内**全无侦测发现到**针对 `node_id` 属性要素的赋值声明输入，网关核心主程序的寻址模块将采取回退策略切换至基于缺省规则运作，它将驱动调用自动配置拼接合成算法从而生成一套类似于格式为 `ns=1;s=当前承载PLC名号标定.测点节点实体命名标定` 这种预置架构模型的自动生成标示信息填补占位（以此规制策略来确保持有旧有数据版本框架描述形式配置文件的系统在完成本次引擎重构升级后依然具备毫无违和顺畅平移的系统层面**全面向后兼容特性承袭**）。

需着重强调指明：此功能的引进定位为对通信规范体系暴露协议映射控制层内的一处**高度可选的、纯业务逻辑侧追加优化强化特性增补动作**；系统面对缺失该 `node_id` 数据声明记录要素的既有传统老式版本配置结构文档体系其总体执行解析运行行为链路无分毫偏差变化。不仅如此，与网关紧耦合配发提供调阅之专属 `ua_monitor` 旁路侧写巡检诊断监测终端应用其同样已全量对齐升格具备针对定制版 NodeId 规则规范模式的深度接轨剖析解码能力特性配置，以确保自底端协议转换中枢至顶层系统性能透视诊测监管面板全流程视窗间能够保持严密维系节点标识读取数据标准的统一度与完全等效同步效果。

| 核心术语概念名词 | 专业释义解读定义与工程所属边界范畴划定 |
|---|---|
| **S7 通信协议族** | 为依托基于 ISO-on-TCP 通信架构规范为底座之上，由西门子企业主导封装构建用以支撑其体系下私有化封闭标准通信的一套底层网络通讯语境准则协议簇规范。 |
| **OPC UA (OPC 统一架构)** | 指代顺应新一代自动化信息交互浪潮，面向工业及制造业控制设备领域主推打造成型的基于跨平台安全开放式架构设计理念的一项体系级核心数据互通框架交互规范 (遵循 IEC 62541 系列核心国际标准体系)。 |
| **Data Block (数据块，通称 DB)** | 特指于西门子 S7 可编程逻辑控制器 (PLC) 的自身架构存储体系模型结构当中，专用于承担诸如承载长期持久化变量驻留存储、或是留存静态配置参数及维持保存各类受控运行过程态暂存控制数据信息的关键性静态结构化主功能逻辑存放资源空间块单元。 |
| **大端序 (Big-Endian 通信存储模型)** | 系在工业领域底层数字信息编码转换通信应用场景里最为司空见惯且屡受推崇的、确立采用将字节信息序列中表征最高量级层级的有效重要权值关键字节位 (MSB 字节) 置于内存较低内存寻址位起始进行优先布局排序驻扎存储规则原则的一项底层计算机架构数字二进制阵列存放流向模型。 |
| **NodeId (通信标识节点句柄)** | 被系统性地在广义 OPC UA 规范地址信息映射分发空间蓝图构型当中，引入赋予用于扮演在全局视域维度下精准独一无二对特定个体化节点通信信息功能对象予以明确识别确认、进而为其调阅访问动作下发统一凭证属性功能标号的一种唯一化基础识别标志化凭据控制句柄参量。 |
| **Quality Code (通信系统数据质量代码标记)** | 是基于在全套 OPC UA 架构约束级规范体例之下，体系内设引入针对所捕获探知数据其取样来源的可靠度真伪程度有效状况特征及信道传输通信链路稳定性表现状态情况等层面进行标准化量化评估衡量测度以及结果附注标示输出功能支持的一项关键辅助评价指标体系基础考核标尺参量数据群。 |
